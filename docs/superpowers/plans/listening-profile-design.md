## Problem Statement

TuneSet(个人 side project,QQ 音乐歌单 AI 分类工具,v0.6.1)新增功能:用户扫码登录 QQ 音乐后,用 L-1124/QQMusicApi 拉用户信息+歌曲信息,前端展示成"听歌画像"。

现有 TuneSet 登录后直奔分类流程,没单独展示用户的 QQ 音乐身份和歌曲富信息。本功能基于"我喜欢"歌单的 `get_labels` 标签数据,做听歌画像可视化(标签云/歌手排行/风格雷达),默认私人+一键分享,差异在即时性+AI 分类联动,10x 方向是 AI 听歌人格。

## What Makes This Cool

- **画像即分类引擎**:画像是分类触发器,不是展示物。AI 看画像识别风格簇后主动建议分类(如"你 8 首民谣藏在电子标签里,要不要单独开歌单"),一键走现有 HITL 分类。这是 TuneSet 独有闭环--Spotify Wrapped / Last.fm / 年度报告都不会帮你整理歌单。
- **复用全部已有栈**:`get_labels` -> 聚合画像 -> AI 调用 -> 分类方案 -> HITL,零新增基础设施。
- **AI 听歌人格**:LLM 基于画像生成"你是 XX 型听众"人格描述,把冷冰冰标签变成故事。
- **即时性**:基于"我喜欢"+官方标签随时生成,不需 scrobble 插件不等年底。

## Constraints

- **数据源**:`get_fav_song`(我喜欢,需 euin)+ `get_labels`(songid 逐首取标签,无批量接口)+ 任意歌单(`SonglistApi.get_detail` 复用 `/api/songlist/shared`)
- **标签缺失**:CLAUDE.md 规定分类时缺标签补 `get_lyric` 歌词(分类依据来源),画像是否 fallback 需权衡(见 Open Questions)
- **前端栈**:React 19 + react-router-dom 7 + dnd-kit + axios + radix-ui + zustand + @tanstack/react-query + tailwindcss + sonner + lucide-react(刚做完品牌化/UI 重构)。新引入可视化库 recharts(雷达图+柱状图)+ react-wordcloud(标签云)需加 `package.json` 依赖。**html2canvas 已弃用**(改分享链接页,见 premise 4)
- **AI 栈**:anthropic + openai 双 SDK,通过 `ai_provider.chat` 调用
- **部署**:个人服务器,docker-compose,SQLite + Redis
- **可视化功底**:单人项目难比大厂年度报告团队(陛下接受此张力)
- **首版数据源(修订)**:**双入口**——"我喜欢" + 任意歌单,均需扫码登录(陛下 D5/D6 定夺)
- **分享持久化(新增)**:公开链接展示的是画像生成时快照,持久化 SQLite(`profile_snapshots` 表,带 `version` 列),非每次访问重拉;原 Redis 缓存仅用于生成期热数据(见"缓存与刷新策略")

## Premises

1. **数据源(修订)**:"我喜欢"(`get_fav_song`)+ 任意歌单(`SonglistApi.get_detail` 复用 `/api/songlist/shared`)+ `get_labels` 标签,够生成三图。**首版双入口**(陛下 D5 定夺),两个入口都需扫码登录(陛下 D6 定夺:身份限流)。
2. **定位张力**:陛下选"类年度报告+分享",但单人项目可视化功底难比大厂;独特性在即时性+AI 联动不在精美。陛下接受此张力不改。
3. **AI 联动(修订)**:画像是分类触发器--AI 看画像识别风格簇,主动建议分类(如"8 首民谣藏在电子标签,要不要开歌单")。复用全部已有栈(`get_labels`->画像->AI->分类->HITL)。
4. **分享(修订)**:首版做**只读分享链接页**——用户主动点"分享"生成随机 token 公开链接,别人持链接在网页看画像快照(含封面)。画像默认私人(陛下 D3 定夺),token 可撤销。**公开链接展示的是生成时快照**(持久化 SQLite,非每次重拉,防打爆 AI 配额)。原 html2canvas 截图方案弃用(陛下 D2 定夺)。
5. **10x 预留(修订)**:AI 听歌人格是 10x 方向。首版**保留最小预留**——`ProfileData` 加可选 `extra: dict` 透传字段(默认空)+ 分享快照表带 `version` 列(持久化本就要建表,顺带零成本);不预先设计多歌单/时间轴/社交对比的抽象(陛下 D4 定夺,与 CLAUDE.md Simplicity First 有张力,陛下拍板优先)。

## 画像数据结构 (ProfileData)

```python
class ProfileData(TypedDict):
    user_id: int
    euin_masked: str               # 脱敏 euin,用于日志;fav 场景=扫码登录用户 euin,任意歌单场景=当前登录用户 euin(非歌单 owner euin,用于审计追溯)
    total_songs: int               # "我喜欢"总歌数
    songs_with_labels: int         # 成功取到标签的歌数(算缺失率)
    tag_counts: list[dict]         # [{tag: str, count: int, dimension: str}],按 count 降序,dimension 标签维度(风格/语种/情绪/场景)
    artist_counts: list[dict]      # [{artist: str, count: int}],Top 10
    style_distribution: dict       # {风格轴: 权重},六轴:流行/摇滚/电子/民谣/说唱/R&B,权重=该轴标签歌曲数/有标签歌曲数
    personality: str               # AI 人格描述(2-3 句)
    style_clusters: list[dict]     # [{cluster_name: str, judge_rule: dict, song_ids: [int], suggestion: str}],AI 识别的风格簇;judge_rule 为结构化表达式(见"风格簇 judge_rule 格式"),后端按规则用内存 per-song 标签匹配 song_ids,不把 per-song 喂 AI(省 token)
    generated_at: str              # ISO 时间戳
    extra: dict                    # ponytail: 10x 预留透传字段(陛下 D4 定夺),默认空;不预设计多歌单/时间轴/社交对比抽象,真要扩时按需填
```

## 风格簇 judge_rule 格式

`judge_rule` 为**结构化表达式**(非自然语言),后端可机械匹配 song_ids。初版格式(实现时按 prompt 调优):

```python
# judge_rule 示例:民谣轴权重>0.15 且该轴标签在 tag_counts 排名>5 的歌
{"axis": "民谣", "axis_weight_gt": 0.15, "tag_rank_gt": 5}
# 或:含某标签但该标签排名低于阈值
{"tag": "电子", "tag_rank_gt": 3}
```

- 后端遍历内存 per-song 标签,按 `judge_rule` 的 `axis`/`tag` + 阈值条件匹配出 `song_ids`。
- AI 只返回 `cluster_name + judge_rule + suggestion`(省 token);若实现时发现结构化 judge_rule 表达力不足,**fallback 退化**:放弃省 token 优化,让 AI 直接返回 `song_ids`(成本略增但实现简单,列为备选)。

## 风格雷达六轴映射规则

QQ 音乐 `get_labels` 返回标签维度为"风格/语种/情绪/场景"。映射规则(初版,实现时验证调整):

- **风格维度标签** -> 直接映射六轴(流行/摇滚/电子/民谣/说唱/R&B)。若标签不在六轴(如"古典""爵士"),归入"其他"不计入雷达,但仍进标签云。
- **语种/情绪/场景维度标签** -> 不映射雷达轴,只进标签云。
- **六轴权重** = 该轴标签出现的歌曲数 / 有标签歌曲总数(归一化 0-1)。

## 限流与成本管控

CLAUDE.md 强制:AI 成本开发者承担,所有用户共用 -> 必须强限流。

- `/api/profile/generate` **独立限流**(`rate_limit_profile_daily`,默认 5 次/日/用户),不复用分类配额,避免画像刷爆挤占分类。双入口("我喜欢"+任意歌单)共享该日限(按 user_id 计,不按 source 拆分)。
- 画像生成涉及 2 次 AI 调用(人格描述 + 风格簇识别),AI 调用走现有审计日志(`log_audit`,action=`profile_generate`),记录 `llm_calls` + `total_cost`。
- 限流配置走 `backend/app/config.py` settings 可配 + 默认值(遵循配置管理规则)。
- **多账号绕过 trade-off**(透明承认):身份限流挡不住用户注册多账号/陛下自多账号刷 AI 配额。缓解:依赖邀请制收紧注册 + `rate_limit_profile_daily=5` 足够低 + 审计监控异常(user_id 短时高频生成告警)。按歌单维度限流成本/复杂度上升,不首版做。

## 缓存与刷新策略

- 画像结果存 Redis(key: `profile:{user_id}:{source_type}:{source_ref}`,TTL 1 小时),避免重复拉取+AI 调用。**注意**:Redis 仅缓存生成期热数据(给已登录用户秒回当前画像);公开分享链接展示的是 SQLite 快照,不走 Redis。**双入口 key 区分**:`source_type=fav` + `source_ref=euin_masked`(我喜欢),或 `source_type=playlist` + `source_ref=歌单id`(任意歌单),避免同一用户对"我喜欢"与"歌单 A"缓存互相覆盖。
- **1h 内**再次请求(同 source):直接返回缓存。
- **手动刷新**:前端"刷新画像"按钮,强制重新拉取 `get_fav_song`/`get_detail` + `get_labels` + AI 调用,覆盖 Redis 缓存**并写新快照到 SQLite**(新 snapshot 行,旧分享 token 仍指向旧快照——分享的是生成时的画像,刷新不影响已分享链接)。
- TTL 1h 内用户加了新歌,画像是过期的--用户可手动刷新。不自动监听歌单变化(避免复杂度)。
- **并发锁**:dispatch 前检查 Redis `profile:lock:{user_id}:{source_type}:{source_ref}`(`SET NX EX`,TTL 与 task 超时一致),防同一用户对**同一 source**并发生成多 task 浪费 AI 配额;task 完成/异常主动 `DEL`。不同 source 不互斥(同用户可并发生成 fav + 歌单 A)。

## 快照与分享 token 生命周期

- **snapshot 清理**:用户每次刷新写新行,旧 token 指向旧 snapshot 不自动删;应用层定期清理(Celery beat 或 on-demand,实现时定)——保留每用户每 source 最近 N 条(N 可配,默认 10),超出的且无有效(未撤销且未过期)token 引用的 snapshot 行删除;撤销 token 时若关联 snapshot 无其他有效(未撤销且未过期)token 引用则可删。避免 SQLite 无限增长。
- **token 熵**:`profile_share_tokens.token` 用 `secrets.token_urlsafe(32)`(256-bit),枚举爆破不可行。
- **token 过期**:token 表加 `expires_at`(默认 90 天,可配),公开路由校验未过期;过期自动失效(用户忘撤销=最多泄露 90 天,非永久)。
- **公开路由响应**:`/api/profile/shared/{token}` 响应 `Cache-Control: no-store`,每次实时查 DB 校验 `revoked_at`/`expires_at`,防 CDN/Nginx 缓存导致撤销后短时仍可访问。
- **公开路由限流**:生成端身份限流不走 IP(见"限流与成本管控"),但**公开只读路由加独立 IP 频率限流**(防爬虫枚举/抓取),与生成端限流分开计数。

## 边界处理

- **歌数 < 10**:前端提示"歌曲太少,至少需要 10 首才能生成画像",不调 AI。
- **标签缺失率 > 50%**(songs_with_labels/total_songs < 0.5):降级展示--仅歌手排行 + 标签云,跳过风格雷达 + AI 人格 + 风格簇(均走 AI,数据不足以支撑,一起跳过)。提示"标签数据不全,已跳过雷达/人格/风格簇"。
- **AI 返回 0 条风格簇**:展示"暂无分类建议,你的听歌口味较均匀"。
- **AI 调用失败**:人格/风格簇降级为空,三图仍展示(三图不依赖 AI)。

## 性能与异步架构

### get_labels 逐首调用性能(可行性硬伤)

`get_labels` 无批量接口,500 首"我喜欢"= 500 次 API 调用。串行约 250s+,且 Celery `task_time_limit` 有限。

**解法**:
- `asyncio.Semaphore(20)` 控制持续并发上限(20 个 slot 始终满,比逐批 gather 快),避免触发 QQ 音乐频率限制。
- 进度通过 SSE 推送(复用现有 `stream.py` SSE 模式,按 euin 临时缓存),用户看到“拉取标签 120/500”。
- Celery task 超时评估:若 500 首并发后仍超 `task_time_limit`,调大画像 task 的独立超时(不复用分类的超时)。
- **超大歌库截断**:config 加 `profile_max_songs=500`(可配),“我喜欢”超 500 首仅取前 500 生成画像,前端提示“仅取前 500 首生成画像”。避免数千首 `get_labels` 超出 `task_time_limit`。

### 同步/异步

`/api/profile/generate` 走 **Celery 异步**(复用现有 `classify_task.delay` 模式):
- 立即返回 `profile_id` + `thread_id`。
- 前端通过 SSE(复用 `stream.py`)拉进度:拉取标签 -> 聚合 -> AI 人格 -> 风格簇。
- 不走 FastAPI 同步端点(会阻塞 worker)。

### AI 调用架构(不走 LangGraph)

画像生成**不走 LangGraph**(无 HITL 循环,interrupt/checkpoint 价值不大),直接调 `ai_provider.chat()`:
- `generate_personality(profile_data) -> str`:三结构序列化成 prompt,生成人格描述。
- `identify_style_clusters(profile_data) -> list[dict]`:基于聚合标签分布生成风格簇+判定规则(如“民谣轴权重>0.15 且 tag_counts 未进前5”)。**不把 per-song 标签喂 AI**(省 token),AI 只返回 `cluster_name + judge_rule(结构化表达式) + suggestion`,后端用内存中 `get_labels` 拉取的 per-song 标签按 judge_rule 自行匹配 song_ids。judge_rule 格式与 fallback 见"风格簇 judge_rule 格式"。

### 分类触发(联动机制时序)

画像生成是独立流程(不走 LangGraph);分类触发是"一键把风格簇歌曲子集传给现有分类":

```
前端 /profile 页面
  |-- 点击风格簇建议"开歌单"按钮
  |-- POST /api/classify/start {song_ids: 风格簇 song_ids, target_dirname: 建议名}
  |-- 复用现有 classify_task.delay() + LangGraph HITL 流程
  |-- SSE 推分类进度
  |-- 分类完成,歌单已创建+添加(song_id 精确)
```

画像 graph 和分类 graph 独立,不共享 state。联动通过 `/api/classify/start` 传 song_ids 实现。

### 分享链接页与快照持久化(替代原 html2canvas 截图方案)

陛下 D2 定夺:首版做**只读分享链接页**,不做 html2canvas 截图。这带来后端持久化与公开路由需求(范围放大,不隐瞒):

- **快照持久化**:画像生成完成后,`ProfileData` 序列化存 SQLite `profile_snapshots` 表(`id` / `user_id` / `source_type`(fav/playlist)/ `source_ref`(euin_masked 或歌单 id)/ `data_json` / `version` / `created_at`)。公开链接展示的是该快照,**非每次访问重拉**(避免打爆 AI 配额 + QQ 音乐频率限制)。snapshot 生命周期(清理/保留)见"快照与分享 token 生命周期"。
- **分享 token**:`profile_share_tokens` 表(`token` 随机串(`secrets.token_urlsafe(32)`,256-bit) / `snapshot_id` / `user_id` / `created_at` / `expires_at`(默认 90 天) / `revoked_at`)。用户在画像页点"分享"生成 token,持 `/profile/shared/{token}` 公开只读访问;点"撤销分享"置 `revoked_at`,链接即失效;过期自动失效。
- **公开只读路由**:`/api/profile/shared/{token}` 走独立路由,**绕过 JWT 鉴权**(仅校验 token 有效、未撤销、未过期——每次实时查 DB),返回快照 `data_json`;响应 `Cache-Control: no-store`;加**独立 IP 频率限流**(防爬虫枚举/抓取,与生成端身份限流分开计数)。前端 `/profile/shared/:token` 路由渲染只读画像(无"开歌单"/"刷新"等操作按钮,纯展示)。
- **隐私**:画像默认私人,无 token 不可公开访问;只有用户主动生成 token 才对外可见,且可随时撤销/过期自动失效(陛下 D3 定夺)。
- **跨域图片**:分享链接页在网页内展示专辑封面,走后端代理 `/api/proxy-image?url=...`(加 `Cache-Control` + 白名单 QQ 音乐域名),避开 canvas tainted(网页 `<img>` 本身不受 CORS 限制,代理仅为统一缓存与可控性)。

> 原 html2canvas 截图方案的跨域 canvas tainted 问题随方案弃用而消失;代理方案转为分享页封面的可选增强(首版可用 QQ 音乐直链 `<img>`,代理列后续)。

## Approaches Considered

### Approach A: 最小可行--纯展示画像

只做画像展示(三图+AI 人格+截图分享),不做分类触发联动。`/profile` 页面扫码后一键生成。Effort S-M(1-2 天),Risk Low。最快出 whoa,但浪费"画像即分类引擎"差异化,纯展示跟大厂比可视化扬短避长。

### Approach B: 理想架构--画像即分类触发器(闭环)

画像展示 + AI 识别风格簇 + 主动建议分类 + 一键走现有 HITL。画像->AI 调用->分类建议->HITL。Effort M-L(4-6 天,含限流/审计/并发/SSE 工程约束),Risk Med。实现独有差异化,复用全部已有栈零新增基建,但首版要做联动逻辑工作量六,AI prompt 需调优。

### Approach C: 创意横向--听歌体检报告

不做通用画像,做"听歌体检":风格偏科/遗忘歌曲/版本重复(`get_other_version`)/缺口建议,报告+处方。Effort M(3-4 天),Risk Med-High。独特品类避开可视化比拼,但偏离"类年度报告"对标,诊断指标需设计。

## Recommended Approach

**Approach B: 画像即分类触发器(闭环)**。陛下修订 premise 3 确认"画像即分类触发器",B 直接实现这个独有差异化且复用全部已有栈(零新增基础设施)。A 最快但浪费陛下刚确认的差异化,C 最创意但偏离 D5 对标。B 最契合陛下刚确认的方向。

## Open Questions

1. **标签缺失是否 fallback `get_lyric`?** CLAUDE.md 规定分类时缺标签补歌词(分类依据来源)。画像若标签缺失率 >50% 是降级展示(见边界处理),还是 fallback `get_lyric` 补标签?subagent 建议跳过(慢),但可能影响画像质量。**建议**:首版降级展示,fallback 列后续(避免 500 首再逐首 `get_lyric` 翻倍耗时)。
2. **风格簇 AI prompt 如何设计?** “8 首民谣藏在电子标签”这类洞察的生成逻辑。需实现时调优 prompt,让 AI 基于 `tag_counts` + `style_distribution` 输出“标签占比异常”或“标签跟主流不同”的簇。**粗略启发式**(实现时验证):某风格轴权重 > 0.15 但对应歌曲在 tag_counts 未进前 5;或某标签在 style_distribution 占比与 tag_counts 排名差异 > 3 位。
3. **`get_labels` 并发是否触发频率限制?** `Semaphore(20)` 持续并发是否被 QQ 音乐限流?需实现时实测,必要时降并发或加退避重试(复用现有 `subscribe` 接口重试模式)。
4. **画像 task 独立 Celery 超时设多少?** 500 首并发 `get_labels` + 2 次 AI 调用,预估 3-5 分钟。**iter4 已定**:`task_time_limit=600`(独立于分类超时),并发锁 TTL 同步设 600s(见"缓存与刷新策略"并发锁节)。
5. **`get_labels` 返回是否含维度字段?** 六轴映射和风格簇判定都假设 `get_labels` 返回标签带维度(风格/语种/情绪/场景)。`docs/qqmusic-api-interfaces.md` 未确认返回结构。**前置阻塞**:实现前先验证。若无维度,**fallback 强化**(不止关键词匹配):准备人工映射表(标签名->维度,覆盖常见标签)近似六轴,放弃语种/情绪/场景区分(全标签进标签云);或降级为不区分维度。纯关键词匹配有歧义("流行摇滚"算流行还是摇滚),映射表更可控。

## 双入口任意歌单「开歌单」联动语义(补充 S2)

任意歌单画像下,风格簇建议的"开歌单"按钮语义需明确:

- **场景**:用户给**别人的**歌单(粘贴分享链接)生成画像,AI 识别出"8 首民谣藏在电子标签"风格簇。点"开歌单"把这 8 首的 song_ids 子集传给 `/api/classify/start`,**开成用户自己的新歌单**(走现有 HITL:AI 分类建议 -> 用户拖拽确认 -> `SonglistApi.add_songs` 按 song_id 精确添加到用户新建歌单)。
- **前提**:用户已扫码登录(有 musickey + euin),`add_songs` 需登录态创建歌单。
- **语义边界**:从他人歌单里挑出风格簇子集,开成自己的歌单——本质是"从他人歌单挖歌"。song_id 精确添加保证版本一致(绕开 QQ 音乐按歌名模糊匹配)。
- **不做的**:不从用户"我喜欢"里反推是否含这些歌(避免跨 source 交叉逻辑);不提示"这些歌你已喜欢"(画像 graph 与分类 graph 独立,不共享 state)。

## Success Criteria

- 扫码登录后,一键生成听歌画像(AI 人格描述 + 标签云 + 歌手 Top10 + 风格雷达)
- **双入口**:对"我喜欢"或任意歌单(粘贴分享链接)均能生成画像(陛下 D5)
- AI 输出 0-N 条风格簇分类建议(0 条时展示"暂无建议,口味较均匀";N 条时每条带"开歌单"按钮)
